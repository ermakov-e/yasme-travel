# Feature-Sliced Design (FSD) — Yasme Travel

## Overview

Feature-Sliced Design (FSD) — архитектурный подход для фронтенд-приложений, обеспечивающий масштабируемость, переиспользуемость и ясные зависимости между слоями.

В Yasme Travel используется строгий FSD с разделением на слои, но внутри фич можно гибко организовать структуру для MVP.

---

## Layers

### 1. app

- Главный слой приложения
- Инициализация, провайдеры, роутинг
- Примеры:
  app/layouts/
  app/router.tsx
  app/styles/

### 2. pages

- Компоненты страниц
- Формируют композицию фич
- Примеры:
  pages/AuthPage/AuthPage.tsx
  pages/MapPage/MapPage.tsx

### 3. widgets

- Крупные UI блоки, состоящие из нескольких компонентов
- Могут использовать features и shared
- Примеры:
  widgets/MapToolbar/

### 4. features

- Пользовательские сценарии, содержащие логику и UI
- Внутренняя структура гибкая, но желательно отделять model/ui
- Примеры:
  features/add-point/
  model/ # бизнес-логика, state, slice
  ui/ # компоненты фичи

### 5. entities

- Бизнес-сущности приложения (User, Group, Point, Photo)
- Включают model и ui для конкретной сущности
- Примеры:
  entities/user/
  entities/group/
  entities/point/

### 6. shared

- Переиспользуемые модули и компоненты
- Подкаталоги:
  shared/ui/ # кнопки, инпуты, модалки
  shared/hooks/ # хуки
  shared/types/ # глобальные типы

- Запрещено:
- дублировать код из других слоев
- использовать цвета вне theme или фиксированные px

---

## Dependencies Rules

- Импорты только вниз или на том же уровне
- Запрещено: feature → app/pages/widgets
- Разрешено: page → feature → entity → shared
- Shared не зависит ни от кого
- Features могут использовать другие features, только если они независимы
- Entities могут использовать shared
- Widgets могут использовать features и shared

---

## Feature Structure Guidelines

- Гибкая структура внутри фич для MVP
- Рекомендуется разделять:
- `model/` — состояние, slice, бизнес-логика
- `ui/` — компоненты
- `lib/` — утилиты, хелперы (по необходимости)
- Не использовать абсолютные пути вне структуры
- Все фичи должны быть self-contained

---

## Naming Conventions

- Папки и файлы в `kebab-case`
- Компоненты React с `PascalCase`
- Slice и модель — `camelCase`
- Shared-компоненты именуются по функции (`Button`, `InputField`)

---

## Best Practices

- Всегда определяй слой перед созданием нового кода
- Не дублируй логику между слоями
- UI должен использовать theme и MUI компоненты
- Компоненты должны быть максимально переиспользуемыми
- Состояние хранить в slice, локальный state — в компоненте
- Рефакторинг допускается только после согласования
